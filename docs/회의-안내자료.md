# 강화유니버스 — 팀 회의 안내자료

> 2026-04-27 작성. 6월 최종 발표 전 디자인·정책 합의를 위한 자료.
> **개발** 이용찬 / **디자인** 김민경 / **기획·데이터** 정채연 / **데이터** 전현승

---

## TL;DR (3줄)

- 지금까지: 참여자 카드 발급 → 도감 → 공개 임팩트 대시보드 → 관리자 운영까지 전부 작동 (노트북에서 클릭하며 시연 가능한 상태).
- 회의에서 결정: **정책 6 + 디자인 7 + 시연 3 = 16개 항목.** 그 결과로 디자인 방향이 잡혀야 다음 단계가 굴러갑니다.
- 그 다음: 회의 결과 반영 → UI 한 번에 정비 → 편지·응원 기능 → 시연 데이터 채우기 → **6월 최종 발표**.

## 1. 지금까지 만든 것 (= 클릭하면 작동하는 것)

5종 사용자, 각자 화면이 분리되어 있습니다.

### 비로그인 / 일반 방문자
- **공개 랜딩**(`/`) — "오늘도 강화도가 조금씩 더 강화됩니다" 인트로 + 최근 카드 + 카테고리 진입
- **강화의 진척**(`/impact`) — 숫자 카운터 5개 + 카테고리별 진척바 + 진행 중 프로젝트 + **환대의 계보도(노드맵)** + 최근 카드
- **프로젝트 리스트/상세**(`/projects`) — 카테고리별 프로젝트 + 진척바 + 회차 타임라인 + 공개 카드
- **가게 리스트/상세**(`/shops`)
- **공개 피드**(`/feed`) — 공개 카드 시간순

### 참여자 (카카오 로그인)
- **QR → 카드 발급**(`/entry/[QR토큰]`) — 사진 + 메모 + 공개 토글 + 초상권 동의
- **내 도감**(`/collection`) — 발급 카드 그리드
- **카드 상세**(`/collection/[id]`) — 본인 앞면 (뒷면=응원·편지는 다음 단계)
- **내 프로필**(`/me`)

### 사장님 (가게 코드 로그인)
- **로그인** + **우리 가게 홈**(자리 표시만) — 다음 단계에 카드 리스트·편지 작성 추가

### 크루 (공용 코드 로그인)
- **크루 대시보드** — 진행 중 회차의 상태(예정→진행→완료) 업데이트, 하이파이브·노트 응원

### 관리자 (청풍)
- **운영 홈** / **프로젝트·회차 관리** / **가게·QR·사장님 코드 관리** / **공개 검수 큐** / **신고 대응**

### 코드로 이미 강제된 정책
- 같은 회차 여러 번 참여 = 카드 여러 장 (반복 허용)
- 카드는 **기본 비공개**. 본인이 공개 토글해야 외부에 노출.
- **좋아요 / 팔로우 / 공개 댓글 / 개인 점수 모두 없음** (게임 규칙)
- 응원은 카드 뒷면에만, 공개·비공개 선택 가능
- 사진 신고·삭제 절차 있음

## 2. 이미 굳어진 것 (회의에서 가급적 유지)

뒤집으면 일정 큰 영향.

| 영역 | 결정 |
|---|---|
| 사용자 종류 | 5종 (비로그인 / 참여자 / 크루 / 사장님 / 관리자) |
| 4개 카테고리 | 환대의 공유지 / 네트워크 / 세계 / 정책 |

| 기기 우선순위 | **데스크탑 1순위**, 모바일 부차 (발표는 프로젝터·노트북) |
| 디자인 톤 (잠정) | "Ink-on-Linen" 도감 — 종이질 배경 + 잉크 + 황토 강조 |

## 3. 회의에서 정해야 하는 것 (16개)

각 항목 = **지금 상태 / 질문 / 결정이 영향 미치는 곳**.

---

### 묶음 ① 정책·거버넌스 (6개)

#### 1-1. 가치 제안 1문장
- **지금**: 비어 있음
- **질문**: 발표 첫 1문장으로 "이 프로젝트는 무엇인가" 어떻게 답할까?
- **영향**: 랜딩 메인 카피, 발표 첫 슬라이드, 모든 외부 자료의 한 줄

#### 1-2. 운영 거버넌스 — 누가 결정·승인하나
다음 5가지 결정권자:
1. 카테고리 정의 / 프로젝트 매핑
2. 회차 생성 / 공개 승인
3. 중복·오입력 정정
4. 사진·메모 신고 대응 / 삭제 처리
5. 계정 분실·재발급 (사장님 코드, 관리자 계정)
- **영향**: 청풍 인계 시 운영 매뉴얼

#### 1-3. 랜딩 라우팅 분기 룰
- **지금**: 로그인하면 자동으로 본인 화면(참여자→도감 / 사장님→가게 / 관리자→운영홈)으로 이동
- **질문**: 이게 맞나? 로그인 사용자도 가끔 공개 랜딩(/)을 보고 싶을 텐데?
- **영향**: 모든 사용자의 첫 진입 경험

#### 1-4. 진척도 기본 기준
- **지금**: 4종 (시간 기반 / 회차 수 / 목표값 / 복합) 중 디폴트 미정
- **질문**: 새 프로젝트를 만들 때 어떤 기준이 가장 흔한가?
- **영향**: 관리자가 새 프로젝트 만들 때 폼의 디폴트 값

#### 1-5. 카테고리 ↔ 프로젝트 매핑 리스트
- **지금**: 비어 있음
- **질문**: 발표에 들어갈 실제 청풍 프로젝트 N개를 4개 카테고리에 어떻게 배치하나?
- **영향**: 시연 데이터, 프로젝트 화면, 카테고리 진척바

#### 1-6. 경쟁/중독 메커니즘 금지 정책
- **지금**: 코드로 이미 금지 (좋아요·팔로우·랭킹·개인 점수 없음)
- **질문**: 유지인가, 일부 (예: 따뜻한 응원 카운트) 허용인가?
- **영향**: 발표 차별점 ("우리는 SNS 가 아니다")

---

### 묶음 ② 디자인 방향 (7개) — 디자이너 주도

> 외부 AI 검토에서 **디자인 약점 16개** 지적됨. 디자이너 결정 이후 한 번에 반영 예정.

#### 2-1. Ink-on-Linen 톤 강도
- **지금**: 종이질 배경 + 잉크 + 황토 강조. 단 카드·섹션 톤이 무난한 편(평범한 카드 모음으로 보일 위험)
- **질문**: 도감 톤을 어디까지 강하게?
  - (A) 카드·섹션·CTA·배지에 잉크 그림자·황토를 일관되게 강하게
  - (B) 지금처럼 부드럽게 유지
- **영향**: 모든 페이지의 카드·버튼·배지 시각

#### 2-2. 데스크탑 화면 폭
- **지금**: 본문 폭 약 1024px. 1440~1920 프로젝터 환경에서 양옆이 비어 보임
- **질문**: 발표 화면을 더 넓게(약 1280px 또는 풀폭) 할까?
- **영향**: 랜딩 / 임팩트 / 프로젝트 / 도감 / 피드 모든 페이지

#### 2-3. /impact 화면 서사 순서
- **지금**: 카운터 → 카테고리 → 진행 프로젝트 → 노드맵 → 최근 카드 (5단 수직 적층, 동등 위계)
- **질문**: 발표 시 청중을 어떤 순서로 끌고 갈까? 어디를 먼저 강조할까?
- **영향**: /impact 섹션 배치, 발표 흐름

#### 2-4. ActivityCard "도감"성 정의
- **지금**: 사진 + 일련번호 + 공개도장. 외부 검토에서 "SNS 카드와 별 차이 없다"는 지적
- **질문**: 도감(=식물도감·우표앨범 같은 컬렉션 책)으로 가려면 어떤 시각 요소를 더할까?
  - 인쇄물 번호 체계 / 표제·부제 / 종이 질감 / 뒷면 암시 ...
- **영향**: 카드 디자인 — **발표의 시각적 핵심**

#### 2-5. 발표용 글자 크기 바닥선
- **지금**: 작은 글자(10~13px)가 자주 사용됨. 1m 거리 1080p 프로젝터에서 거의 안 보임
- **질문**: 발표 페이지 최소 글자 크기를 얼마로?
- **영향**: 거의 모든 화면

#### 2-6. 노드맵 발표 모드
- **지금**: 자유 탐색(드래그·줌·hover로 상세 표시)
- **질문**: 발표 시
  - (A) 단계별 자동 하이라이트(스토리 모드) 따로 만들까
  - (B) 자유 탐색 그대로 가고 시연자가 라이브로 클릭할까
- **영향**: /impact 의 환대의 계보도

#### 2-7. 공개 카드 상세 페이지 (`/feed/[id]`)
- **지금**: 본인 카드만 상세 진입 가능. 비로그인은 공개 카드를 클릭해서 자세히 보지 못함
- **질문**: 비로그인·타 참여자도 공개 카드 1장을 자세히 열어볼 수 있게 할까?
- **영향**: 발표에서 "카드 → 집계 → 사례" 왕복 가능 여부

---

### 묶음 ③ 시연 시나리오 (3개) — 기획 주도

#### 3-1. 발표 데모 클릭 흐름
- **지금**: 비어 있음
- **질문**: 발표 5~10분 동안 어떤 순서로 어떤 화면을 보여줄까?
- **영향**: 화면 우선순위, 데이터 채울 우선순위

#### 3-2. 시연용 데이터 양
- **지금**: 가게 1개, 프로젝트 1개 정도. 발표하면 빈 화면이 자주 보임
- **질문**: 카테고리 / 프로젝트 / 회차 / 가게 / 공개 카드를 각 몇 개씩 채울까?
- **영향**: 시연 데이터 시드 작업량

#### 3-3. 발표 모드 우회 방식
시연자 계정으로 로그인된 상태에서 발표 중 공개 랜딩을 보여주는 방법:
- (A) URL 에 `?present=1` 추가 (이미 구현)
- (B) 시크릿 브라우저 + 로그아웃
- (C) 별도 발표 모드 경로 만들기
- **질문**: 어느 쪽이 발표 진행자에게 가장 안전한가?
- **영향**: 발표 진행 가이드

## 4. 외부 AI 검토에서 나온 우려 (디자이너·기획자 참고)

회의 전에 코드를 외부 AI(코덱스)에게 비판적으로 검토하게 했고, 16개 약점을 지적받았습니다. 디자인 결정 이후 일괄 반영 예정. 회의에서 의견 수렴이 필요한 큰 줄기:

| 영역 | 우려 |
|---|---|
| 화면 폭 | 데스크탑인데 양옆이 비어 보임 |
| 톤 일관성 | 도감 톤이 약하고 무난한 카드 모음으로 보임 |
| 한글 친화 라벨 | "한글 중심" 선언했지만 영문 태그(No., LIVE, DONE)가 자주 보임 |
| /impact 정보 적재 | 한 페이지에 5섹션이 동등하게 쌓여 있음 |
| 카운터 차별화 | 5개가 비슷하게 보여 어떤 숫자가 핵심인지 안 들어옴 |
| ActivityCard | 도감보다 SNS 카드처럼 보임 |
| 글자 크기 | 발표 환경에서 작은 글자가 안 보임 |
| 카드 진입 차단 | 공개 카드 자세히 보기가 막혀서 사례 강조가 끊김 |
| 노드맵 | 멋지지만 한글 톤·접근성·발표 안정성 면에서 위반 가장 많음 |
| Header | 약해서 페이지마다 같은 제품 인상이 희미함 |
| 시연 데이터 | 빈 상태가 그대로 발표에 노출될 위험 |

## 5. 회의 후 작업 순서 (= 6월 발표까지)

```
[5월 1주]  회의 결과 → 플랜 문서 0번 칸에 박음
[5월 1~2주] 디자인 시스템 강화 + 데스크탑 폭·서사 재배치 (UI 한 사이클)
[5월 2~3주] 편지·응원 기능 (사장님 편지 LLM 초안 + 카드 뒷면 + 크루 응원 확장)
[5월 4주]   공개 카드 상세 페이지, 노드맵 발표 모드
[6월 1주]   시연 데이터 풀 스코프로 채우기 + 디자인 다듬기
[6월 2주]   리허설 + prod 배포 + 카카오 도메인 등록
[6월 3주]   최종 발표
```

여유 거의 없음 — **회의 결과가 5월 1주 안에 다 들어와야 안정적**.

## 6. 회의 진행 가이드 (제안)

회의 1회 90~120분 가정. 효율적 순서:

| 시간 | 내용 |
|---|---|
| 5분 | **현황 브리핑** — 이 자료 1·2번 같이 보기 |
| 15분 | **묶음 ① 정책 6개** — 빠르게 의사결정. 미루지 말기 |
| 40분 | **묶음 ② 디자인 7개** — 디자이너 주도, 화면 캡처 같이 보면서 합의 |
| 15분 | **묶음 ③ 시연 3개** — 기획·발표 진행자 주도 |
| 10분 | **외부 AI 검토 결과** 같이 훑고 디자인 묶음으로 흡수할 부분 합의 |
| 10분 | **다음 행동·기한 정리** |

각 항목 결정 즉시 회의록에 적기. 결과를 그대로 `docs/회의-반영-계획.md` 의 0번 칸에 박는 흐름.
