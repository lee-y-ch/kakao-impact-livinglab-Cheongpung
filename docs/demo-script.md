# 강화유니버스 회의 시연 대본

> 목적: 청풍 회의와 팀 회의에서 현재 초기 모델을 같은 순서로 보여주고, 기능 평가보다 방향성 피드백을 안정적으로 수집한다.
> 기준: `docs/roadmap-to-launch.md`, `docs/test-scenarios.md`

## 0. 시연 목표

- "오늘도 강화도가 조금씩 더 강화됩니다"라는 한 줄 정체성이 화면 흐름에서 전달되는지 확인한다.
- 참여자, 강화도 공개 뷰, 운영자/크루/사장님 3개 주인공이 명확히 구분되는지 확인한다.
- 참여자 카드 생성 -> 내 도감 -> 공개 임팩트 집계 -> 사장님 편지로 이어지는 핵심 서사가 설득력 있는지 확인한다.
- 6월 최종 발표 전에 반드시 고칠 것과 발표 이후로 미뤄도 되는 것을 나눈다.

## 1. 준비물

- 배포 URL: `https://ganghwa-universe.vercel.app`
- 관리자 계정: 별도 안전한 채널로 공유한다.
- 크루 공용 코드: 별도 안전한 채널로 공유한다.
- 사장님 테스트 가게 주소와 코드: 별도 안전한 채널로 공유한다.
- 참여자 테스트 계정: 카카오 로그인 가능한 계정 또는 이미 로그인된 브라우저 세션을 준비한다.
- 백업: 배포 장애에 대비해 주요 화면 스크린샷 또는 짧은 녹화본을 준비한다.

| 피드백 정리 | 10분 | 발표 전 필수 수정 범위 확정 |

## 3. 시연 순서

### 3.1 공개 랜딩

1. `/`에 접속한다.
2. 강화유니버스 한 줄 정체성과 현황 카드 3개를 보여준다.
3. 공개 카드 갤러리와 프로젝트 진입 CTA를 보여준다.

확인 질문:

- 첫 화면만 보고 "무엇을 하는 서비스인지" 말할 수 있는가?
- 현황 숫자가 성과처럼 보이는가, 단순 카운터처럼 보이는가?

### 3.2 공개 임팩트

1. `/impact`로 이동한다.
2. 누적 카드, 연결된 가게, 참여자, 에피소드 숫자를 보여준다.
3. 노드맵에서 project/shop/participant/category 관계를 보여준다.
4. 노드 hover 또는 focus 시 관련 카드 미리보기를 보여준다.

확인 질문:

- 이 화면이 "강화도의 관계가 쌓인다"는 메시지를 충분히 보여주는가?
- 노드맵이 너무 복잡하거나 설명이 필요한가?

### 3.3 프로젝트와 가게 둘러보기

1. `/projects`에서 2026 기준 4개 카테고리를 보여준다.
2. 대표 프로젝트 상세로 들어가 에피소드 타임라인과 공개 카드를 보여준다.
3. `/shops`에서 참여 가게 목록을 보여준다.
4. 대표 가게 상세에서 공개 카드와 분위기를 확인한다.

확인 질문:

- 카테고리 이름과 프로젝트 묶음이 청풍의 실제 언어와 맞는가?
- 회의에서 꼭 보여줘야 할 프로젝트나 가게가 빠져 있는가?

### 3.4 참여자 카드 생성

1. `/entry/[qr_token]` 또는 준비된 QR로 카드 작성 화면에 진입한다.
2. 메모 또는 사진을 입력하고 공개 여부를 선택한다.
3. 저장 후 `/collection`으로 이동해 새 카드가 내 도감에 쌓이는 것을 보여준다.
4. `/collection/[id]`에서 카드 상세와 reactions 영역을 보여준다.

확인 질문:

- 참여자가 이 행동을 "미션 수행"보다 "강화에 환대를 남기는 일"로 이해할 수 있는가?
- 공개/비공개 선택 문구가 충분히 명확한가?

### 3.5 사장님 편지

1. `/owner/login`에서 가게 주소와 사장님 코드를 입력한다.
2. `/owner`에서 본인 가게에 연결된 참여자 카드를 확인한다.
3. `/owner/letters/new?activity_id=...`로 이동한다.
4. AI 첫 문장 제안 버튼을 눌러 초안을 만든다.
5. 초안을 그대로 보내지 않고 사장님이 수정해 편지를 저장한다.
6. 참여자 카드 상세에서 편지가 reaction으로 표시되는지 보여준다.

확인 질문:

- "AI 초안, 꼭 수정해서 보내세요"라는 운영 원칙이 잘 전달되는가?
- 사장님 입장에서 편지를 쓰는 부담이 줄어드는가?

### 3.6 크루 운영

1. `/crew/login`에서 공용 코드로 로그인한다.
2. `/crew`에서 예정/진행/완료 에피소드를 확인한다.
3. 에피소드 status를 변경하고 `/projects/[slug]`에 반영되는지 보여준다.
4. 최근 카드에 하이파이브 또는 note reaction을 남긴다.

확인 질문:

- 크루가 현장에서 직접 업데이트할 수 있을 정도로 단순한가?
- 어떤 작업은 크루가 아니라 청풍 관리자만 해야 하는가?

### 3.7 관리자 운영

1. `/admin/login`에서 관리자 계정으로 로그인한다.
2. `/admin` 운영 홈의 검수/신고/진행 요약을 보여준다.
3. `/admin/projects`에서 프로젝트와 에피소드 관리 위치를 보여준다.
4. `/admin/shops`에서 가게 등록, QR token, 사장님 코드 발급 위치를 보여준다.
5. `/admin/review`, `/admin/reports`에서 공개 검수와 신고 처리 흐름을 보여준다.

확인 질문:

- 청풍 담당자가 직접 운영할 수 있는 범위가 어디까지인가?
- 계정 분실, 신고 처리, 데이터 정정의 책임자를 누구로 둘 것인가?

## 4. 장애 시 대체 흐름

| 상황 | 대체 시연 |
| --- | --- |
| 카카오 로그인이 실패함 | 이미 로그인된 브라우저 세션 또는 준비된 `/collection` 화면으로 진행 |
| LLM 초안 생성이 실패함 | 수동 작성 흐름은 막히지 않는다는 점을 보여주고, API key/무료 tier 한계를 설명 |
| 배포 URL 접속이 불안정함 | 로컬 서버 또는 준비된 스크린샷/녹화본으로 핵심 흐름 설명 |
| 노드맵이 복잡해 보임 | 공개 카드가 쌓이며 관계가 만들어지는 데이터 구조를 먼저 설명 |
| 사장님 코드가 틀림 | 관리자에서 재발급 가능한 위치만 보여주고 실제 코드는 별도 채널에서 확인 |

## 5. 회의 중 기록 양식

| 구분 | 내용 |
| --- | --- |
| 관찰 | 사용자가 실제로 멈칫하거나 헷갈린 지점 |
| 질문 | 회의에서 답이 필요한 제품/운영 질문 |
| 결정 | 발표 전 반영할지, 발표 후로 미룰지 |
| 담당 | 청풍 / 팀원 / 개발 |
| 기한 | 회의 후 언제까지 처리할지 |

## 6. 회의 종료 전 반드시 정할 것

- 발표 전 필수 수정 항목
- 발표 이후로 미룰 항목
- 실제 운영 책임자와 권한 범위
- 최종 발표에서 보여줄 대표 프로젝트, 대표 가게, 대표 카드
- 청풍이 직접 수정해야 하는 데이터와 개발자가 수정해야 하는 데이터의 경계
