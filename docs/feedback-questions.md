# 강화유니버스 피드백 질문지

> 목적: 청풍 회의와 팀 회의에서 막연한 감상이 아니라 발표 전 결정 가능한 피드백을 받는다.
> 사용법: 각 질문은 `발표 전 필수`, `발표 후 개선`, `보류`, `결정 완료` 중 하나로 분류한다.

## 1. 핵심 서사

- "오늘도 강화도가 조금씩 더 강화됩니다"가 프로젝트의 한 줄 정체성으로 적절한가?
- 첫 화면만 보고 참여자, 사장님, 크루/관리자가 무엇을 하는 서비스인지 이해되는가?
- 참여자 카드가 발표 내러티브의 중심으로 충분히 보이는가?
- `/impact`가 "강화도의 보이지 않는 성과"를 보여주는 공개 화면으로 설득력 있는가?
- 서비스가 이을랩 로컬유니버스 앱과 충분히 다른 역할을 갖는가?

## 2. 카테고리와 프로젝트 데이터

- 2026 기준 4개 카테고리 이름을 그대로 확정해도 되는가?
  - 액티브 라이프
  - 로컬 문화 공동 창작
  - 글로벌 & 로컬 네트워크
  - 테크 & 솔루션
- 각 프로젝트가 올바른 카테고리에 들어가 있는가?
- 최종 발표에서 반드시 보여줘야 할 프로젝트는 무엇인가?
- 최종 발표에서 빼도 되는 프로젝트나 아직 공개하면 안 되는 프로젝트가 있는가?
- 프로젝트 설명, 에피소드 이름, 가게 설명 중 청풍 언어로 다듬어야 할 문구가 있는가?
- 시드 데이터의 카드 수와 편지 수가 발표용으로 충분한가?

## 3. 참여자 플로우

- QR 진입 후 카드 작성 화면이 충분히 단순한가?
- 사진, 메모, 공개 여부 선택이 참여자에게 자연스럽게 이해되는가?
- 공개 opt-in 정책이 충분히 명확한가?
- `/collection`이 "내 도감"으로 느껴지는가, 단순 목록처럼 느껴지는가?
- 내 도감의 상단 카드, 상세 카드, reaction 영역 중 가장 중요한 화면은 무엇인가?
- 참여자가 반복 참여해 여러 장의 카드를 만드는 흐름이 자연스러운가?

## 4. 사장님 플로우

- 가게 주소와 8자리 코드로 로그인하는 방식이 사장님에게 설명 가능한가?
- 사장님 코드 입력 시 보이기/숨기기 토글이 충분한가?
- `/owner`에서 "내 가게에 온 참여자 카드"만 보는 구조가 이해되는가?
- AI 첫 문장 제안 버튼이 사장님에게 도움이 되는가?
- "AI 초안, 꼭 수정해서 보내세요" 문구가 충분히 강한가?
- 편지는 공개/비공개 중 어떤 기본값이 적절한가?
- 사장님이 실제로 편지를 자주 쓸 수 있도록 더 줄여야 할 입력이 있는가?

## 5. 크루와 관리자 운영

- 크루가 에피소드 상태를 바꾸고 reaction을 남기는 역할이 적절한가?
- 관리자 화면에서 프로젝트, 에피소드, 가게, QR, 사장님 코드 발급 위치가 찾기 쉬운가?
- 공개 검수 큐와 신고 큐를 청풍 담당자가 직접 운영할 수 있는가?
- 관리자가 실수로 공개하면 안 되는 데이터를 공개할 위험이 있는가?
- 운영자에게 필요한 최소 교육 시간은 어느 정도인가?

## 6. 공개 임팩트와 노드맵

- `/impact`의 숫자 지표 중 발표에서 가장 강조할 값은 무엇인가?
- 노드맵이 관계를 보여주는 데 도움이 되는가?
- hover 또는 tap 카드 미리보기가 필요한 정보를 충분히 주는가?
- 모바일에서 노드맵은 유지해야 하는가, 요약형 카드로 바꿔도 되는가?
- 공개 카드가 적을 때와 많을 때 각각 어떤 화면이 더 설득력 있는가?

## 7. 정책과 거버넌스

- 개인 점수, 공개 랭킹, 좋아요, 팔로우, 공개 댓글을 만들지 않는 원칙에 동의하는가?
- 공개 interaction은 reactions, 검수 큐, 필터 정도로 제한하는 것이 맞는가?
- 얼굴이 나오는 사진의 동의 체크와 신고/삭제 흐름이 충분한가?
- 신고된 카드는 누가, 몇 시간 안에 처리해야 하는가?
- 사장님 코드 분실, 크루 코드 노출, 관리자 계정 분실 시 누구에게 연락해야 하는가?

## 8. 최종 발표 구성

- 발표에서 직접 조작할 화면과 스크린샷으로만 보여줄 화면을 어떻게 나눌 것인가?
- 발표 시간이 5분이면 어떤 화면을 잘라야 하는가?
- 발표 시간이 10분이면 어떤 화면까지 보여줄 것인가?
- 청풍 측이 발표에서 꼭 말하고 싶은 문장은 무엇인가?
- 팀원이 맡을 설명 파트와 개발자가 맡을 조작 파트는 어떻게 나눌 것인가?
